
## Q2 — 图传帧数据内存管理

### Part A — 帧缓冲池设计

#### 1. 设计思路
- 程序启动时预分配固定数量、固定大小的内存块（例如 10 块，每块 1MB）。
- 内部维护一个空闲块队列，借出时从队列取出，归还时压回。
- 使用互斥锁保护队列的并发访问，保证线程安全。
- 提供非阻塞 `tryAcquire()` 和阻塞 `acquire()` 两种借出方式。
- 采用 RAII 思想：借出的帧用 `std::unique_ptr` 包装，自定义删除器自动归还，避免内存块丢失。

#### 2. 核心类定义（C++ 接口）

```cpp
#include <cstdint>
#include <memory>
#include <mutex>
#include <condition_variable>
#include <queue>
#include <vector>

// 单个帧的封装
class Frame {
public:
    explicit Frame(size_t size) : data(size) {}
    std::vector<uint8_t> data;
    // 可扩展时间戳、序号等元数据
};

// 帧缓冲池
class FramePool {
public:
    // 构造：预分配 num_frames 个帧，每个帧 buffer_size 字节
    FramePool(size_t num_frames, size_t buffer_size);

    // 非阻塞借出，若无空闲返回 nullptr
    Frame* tryAcquire();

    // 阻塞借出，直到有空闲帧
    Frame* acquire();

    // 归还帧（通常由智能指针自动调用）
    void release(Frame* frame);

    // 获取当前空闲帧数量（调试用）
    size_t freeCount() const;

private:
    std::vector<Frame> pool_;          // 实际存储所有帧
    std::queue<Frame*> free_list_;     // 空闲帧指针队列
    mutable std::mutex mtx_;           // 保护 free_list_
    std::condition_variable cv_;       // 用于阻塞等待
};

// 自动归还的帧指针类型
using FramePtr = std::unique_ptr<Frame, std::function<void(Frame*)>>;

// 扩展 FramePool，返回自动归还的智能指针
FramePtr FramePool::acquireAuto() {
    Frame* f = acquire();
    return FramePtr(f, [this](Frame* p){ release(p); });
}
```

#### 3. 关键实现伪代码

**构造函数**：
```cpp
FramePool::FramePool(size_t num_frames, size_t buffer_size) {
    pool_.reserve(num_frames);
    for (size_t i = 0; i < num_frames; ++i) {
        pool_.emplace_back(buffer_size);
        free_list_.push(&pool_.back());
    }
}
```

**tryAcquire**（非阻塞）：
```cpp
Frame* FramePool::tryAcquire() {
    std::lock_guard<std::mutex> lock(mtx_);
    if (free_list_.empty()) return nullptr;
    Frame* f = free_list_.front();
    free_list_.pop();
    return f;
}
```

**acquire**（阻塞）：
```cpp
Frame* FramePool::acquire() {
    std::unique_lock<std::mutex> lock(mtx_);
    cv_.wait(lock, [this]{ return !free_list_.empty(); });
    Frame* f = free_list_.front();
    free_list_.pop();
    return f;
}
```

**release**：
```cpp
void FramePool::release(Frame* frame) {
    {
        std::lock_guard<std::mutex> lock(mtx_);
        free_list_.push(frame);
    }
    cv_.notify_one();  // 唤醒等待的 acquire
}
```

### Part B — 模拟图传数据流

#### 1. 整体架构
- **接收线程（生产者）**：不断从池中借出空闲帧，模拟写入数据（`sleep`），然后将帧放入线程安全队列。
- **处理线程（消费者）**：从队列中取出帧，模拟处理（`sleep`），最后将帧归还到池中。
- 使用**原子标志**控制线程退出，确保所有资源正确回收。

#### 2. 线程安全队列（阻塞队列）

```cpp
template<typename T>
class BlockingQueue {
public:
    void push(T value) {
        std::lock_guard<std::mutex> lock(mtx_);
        queue_.push(std::move(value));
        cv_.notify_one();
    }

    T pop() {
        std::unique_lock<std::mutex> lock(mtx_);
        cv_.wait(lock, [this]{ return !queue_.empty() || stopped_; });
        if (stopped_ && queue_.empty()) return T();
        T value = std::move(queue_.front());
        queue_.pop();
        return value;
    }

    void stop() {
        std::lock_guard<std::mutex> lock(mtx_);
        stopped_ = true;
        cv_.notify_all();
    }

private:
    std::queue<T> queue_;
    std::mutex mtx_;
    std::condition_variable cv_;
    bool stopped_ = false;
};
```

#### 3. 工作线程实现

```cpp
std::atomic<bool> stop_flag(false);
FramePool pool(10, 1024 * 1024);          // 10个1MB帧
BlockingQueue<Frame*> processing_queue;   // 待处理队列

// 接收线程
std::thread receiver([&] {
    while (!stop_flag) {
        Frame* frame = pool.acquire();     // 阻塞获取空闲帧
        // 模拟写入帧数据（耗时5ms）
        std::this_thread::sleep_for(std::chrono::milliseconds(5));
        // 传递给处理线程
        processing_queue.push(frame);
    }
});

// 处理线程
std::thread processor([&] {
    while (!stop_flag) {
        Frame* frame = processing_queue.pop();
        if (frame == nullptr) break;       // 停止时可能返回空
        // 模拟处理帧（耗时10ms）
        std::this_thread::sleep_for(std::chrono::milliseconds(10));
        // 处理完毕，归还到池
        pool.release(frame);
    }
});
```

#### 4. 优雅退出

```cpp
// 主线程等待用户输入（或按ESC等）
std::cout << "Press Enter to stop...\n";
std::cin.get();

// 通知停止
stop_flag = true;

// 唤醒可能阻塞的线程
processing_queue.stop();

// 等待线程结束
receiver.join();
processor.join();

// 此时所有帧均已归还，pool 析构自动释放内存
```

## 一、架构设计思维

这道题的核心是设计一个**高效、安全、可扩展**的帧缓冲池及模拟数据流。我的设计体现了以下架构思维：

### 1. 分层与解耦
- **缓冲池层（FramePool）**：只负责内存块的分配、回收、线程安全，不关心数据内容。
- **数据流层（接收线程 + 处理线程 + 阻塞队列）**：负责生产者和消费者的协作，与缓冲池通过明确的接口（`acquire`/`release`）交互。
- **好处**：各层职责单一，可独立测试和替换。例如，可以轻松把阻塞队列换成无锁队列，而不影响缓冲池。

### 2. 资源预分配与复用
- 程序启动时一次性申请所有内存块（`vector<Frame> pool_`），避免了运行时频繁 `new/delete`。
- 通过空闲队列（`free_list_`）复用已分配的内存，符合“池化”思想，是性能优化的常见模式。

### 3. 接口设计的合理性
- 提供**阻塞**（`acquire`）和**非阻塞**（`tryAcquire`）两种借出方式，适应不同场景：
  - 接收线程需要持续获取帧，阻塞等待是自然的。
  - 某些模块可能只想尝试借用，若没有则做其他工作，非阻塞更合适。
- 使用**智能指针 + 自定义删除器**（`FramePtr`）自动归还，让用户无需手动调用 `release`，降低出错概率。

### 4. 并发模型的选择
- 采用经典的**生产者-消费者模式**，通过线程安全的阻塞队列解耦两个线程。
- 生产者和消费者只依赖队列，不直接交互，易于扩展（例如增加多个消费者）。

## 二、资源生命周期的理解

资源生命周期包括：**创建 → 使用 → 归还/销毁**。本题中，资源主要是内存块（帧缓冲）。以下是我如何管理生命周期的：

### 1. 创建阶段
- 在 `FramePool` 构造函数中，一次性创建所有 `Frame` 对象并存入 `pool_`，同时将每个 `Frame` 的指针放入空闲队列。
- 所有内存块的**所有权**归属于 `FramePool` 对象（`vector<Frame>` 持有实际内存）。

### 2. 使用阶段
- 使用者通过 `acquire()` 从池中“借出”一个空闲帧的指针。
- 此时，该帧被标记为“已借出”（逻辑上从空闲队列中移除），但内存本身仍由 `FramePool` 管理，使用者只有使用权，没有所有权。
- 为了确保使用者不会忘记归还，我设计了 `FramePtr`（`unique_ptr` 配合自定义删除器）。当 `FramePtr` 离开作用域时，自动调用 `release()` 将帧归还。这利用了 **RAII**（资源获取即初始化）思想，将资源的释放责任与对象的生命周期绑定。

### 3. 归还阶段
- 显式调用 `release(frame)` 将帧指针放回空闲队列，并唤醒可能正在等待的线程。
- 或者通过 `FramePtr` 自动归还。

### 4. 销毁阶段
- 当 `FramePool` 对象析构时，`vector<Frame>` 自动释放所有帧内存。
- 此时要求所有借出的帧必须已经归还（否则会出现悬空指针，但通过 `FramePtr` 可以保证在离开作用域时归还）。在多线程退出流程中，我们通过 `stop_flag` 和 `BlockingQueue::stop()` 确保所有线程退出前，所有帧都已处理完毕并归还。

### 总结：生命周期管理的关键点
- **所有权明确**：`FramePool` 始终拥有内存，使用者只借不拿。
- **自动归还机制**：RAII + 智能指针，避免人工遗漏。
- **退出时清理**：等待所有工作线程结束，确保没有未归还的帧。

## 三、多线程并发如何体现与理解

题目要求“接收和处理需要并发/多线程执行”，并且缓冲池操作必须线程安全。我的设计体现了以下并发要点：

### 1. 线程安全的缓冲池
- 使用 `std::mutex` 保护空闲队列 `free_list_`。
- `tryAcquire` 和 `release` 内部加锁，保证同一时刻只有一个线程修改队列。
- 使用 `std::condition_variable` 实现阻塞等待：当没有空闲帧时，`acquire` 会释放锁并睡眠，直到 `release` 中 `notify_one` 唤醒。

### 2. 生产者-消费者线程模型
- **接收线程（生产者）**：持续调用 `pool.acquire()` 获取帧，写入模拟数据后放入 `BlockingQueue`。
- **处理线程（消费者）**：从 `BlockingQueue` 取帧，处理完后调用 `pool.release()`。
- 两个线程通过队列解耦，并发执行，互不阻塞（除了队列为空时消费者等待）。

### 3. 线程安全的阻塞队列
- `BlockingQueue` 内部同样使用 `mutex + condition_variable` 实现 `push` 和 `pop`。
- `pop` 在队列为空时阻塞，直到有数据或队列被 `stop()`。

### 4. 优雅退出机制
- 使用 `std::atomic<bool> stop_flag`，两个线程循环检查。
- 主线程设置 `stop_flag = true` 后，调用 `processing_queue.stop()` 唤醒可能阻塞在 `pop` 上的消费者线程。
- 消费者收到 `stop` 后，如果队列为空则返回空指针，线程退出。
- 接收线程在 `stop_flag` 为真时，也会退出循环。
- 最后主线程 `join` 等待两个线程结束，确保资源安全释放。

### 5. 理解并发中的常见问题及解决
- **竞态条件**：通过互斥锁保护共享数据（空闲队列、阻塞队列）避免。
- **死锁**：通过确保锁的获取顺序一致（这里只有一个锁，无死锁风险）以及条件变量的正确使用避免。
- **资源泄漏**：通过 RAII 和退出机制确保所有借出的帧最终归还。
- **虚假唤醒**：`condition_variable` 的 `wait` 使用循环条件（`wait(lock, predicate)`）来避免。
