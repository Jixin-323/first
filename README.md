# RoboMaster 操作员客户端 - 招新面试作答项目
本项目为华南农业大学Taurus战队视觉组前后端开发方向招新面试题**Q1 自定义客户端界面开发**的完整作答实现，基于 Qt Quick 6 开发，完整还原 RoboMaster 机甲大师赛操作员客户端的核心功能与交互逻辑。

> 项目开发周期：10天左右，用AI（豆包+deepseek）辅助边做边学，完成全功能开发与调试。

---

## 项目概述
本项目针对 RoboMaster 高校赛的实际比赛场景，开发了一套叠加在机器人第一视角图传画面上的操作员客户端界面，实现了以下核心能力：
1.  比赛全局状态、双方机器人血量、己方机器人核心数据的实时可视化展示
2.  符合官方简化通信协议的鼠标控制指令下发（移动速度、滚轮、左/右/中键状态）
3.  底盘性能、发射模式的快速切换控制，以及比赛暂停/恢复功能
4.  子线程模拟比赛数据实时变化，主线程UI无卡顿响应
5.  适配比赛紧张场景的视觉动效与交互优化，关键数据一眼可辨

项目完全遵循题目要求，使用 CMake 构建工程，无真实网络通信依赖，完整跑通了「数据模拟-界面渲染-指令下发-控制台反馈」的全链路。

## 项目环境与编译运行
### 依赖环境
| 依赖项 | 版本要求 |
|--------|----------|
| Qt     | 6.8.0 及以上（需包含 Qt Quick 模块） |
| CMake  | 3.16 及以上 |
| 编译器 | Windows: MSVC 2022 / MinGW 11.2+；Linux/macOS: GCC 11+ |

### 编译与运行步骤
#### Windows 平台（Qt Creator）
1.  打开 Qt Creator，点击「打开文件或项目」，选择本项目的`CMakeLists.txt`
2.  选择合适的编译套件（Qt 6.8.0 对应的MSVC/MinGW），完成项目配置
3.  点击左下角「运行」按钮，即可自动编译并启动程序

#### 命令行编译（全平台通用）
1.  确保 Qt 环境变量已配置，`qmake`/`cmake` 可在命令行访问
2.  进入项目根目录，执行以下命令：
    ```bash
    # 创建构建文件夹
    mkdir build && cd build
    # 生成构建文件
    cmake ..
    # 编译项目
    cmake --build .
    # 运行程序（Windows）
    ./Debug/appRM.exe
    # 运行程序（Linux/macOS）
    ./appRM
    ```

## 项目结构说明
```
RM-Operator-Client/
├── CMakeLists.txt          # 项目CMake构建配置（符合题目要求，不使用QMake）
├── main.cpp                # 程序入口，初始化Qt应用、QML引擎、C++桥接类
├── rmbridge.h              # 核心桥接类头文件，定义QML可访问的属性与槽函数
├── rmbridge.cpp            # 桥接类实现，包含控制逻辑、数据接收、指令打印、线程管理
├── Main.qml                # 主界面QML文件，所有UI元素、交互逻辑、动画效果实现
├── res.qrc                 # Qt资源文件，管理图传背景图片等静态资源
└── image/              # 图片资源
     └── sb.png          # RoboMaster第一视角图传背景图
```

### 核心文件职责说明
1.  **`rmbridge.h/cpp`**：项目核心C++后端，承担三大职责
    - 数据桥接：通过`Q_PROPERTY`将C++数据暴露给QML界面，实现双向绑定
    - 线程管理：启动子线程`DataGenerator`模拟比赛数据，保证UI线程不卡顿
    - 指令处理：接收QML的鼠标、按键、模式切换操作，输出控制台日志，实现题目要求的操作反馈
2.  **`Main.qml`**：前端UI实现，完全遵循Qt Quick声明式语法，所有UI元素叠加在图传背景之上，保留核心视野完整性
3.  **`main.cpp`**：程序入口，规范初始化Qt应用，将C++桥接类实例暴露给QML上下文，保证前后端通信链路通畅
4.  **`CMakeLists.txt`**：工程构建配置，适配Qt 6标准构建流程，跨平台兼容

## 核心作答思路与架构设计
### 整体架构设计思路
针对题目要求与RoboMaster实际比赛场景，我采用了**「UI主线程 + 数据子线程」的分层架构**，完全遵循Qt的线程安全规范：
1.  **主线程（GUI线程）**：只负责QML界面渲染、用户交互事件接收，不做任何耗时计算，保证界面永远流畅不卡顿，符合比赛场景的低延迟要求
2.  **子线程（数据线程）**：独立运行定时器，每秒生成模拟比赛数据，通过Qt信号槽将数据发送给主线程，完全不接触UI对象，避免了跨线程操作UI的崩溃风险
3.  **桥接层设计**：`RMBridge`类作为前后端通信的唯一桥梁，封装了所有QML可访问的属性和槽函数，实现了界面与逻辑的解耦，代码可维护性更高

### 核心模块设计逻辑
#### 1. 数据绑定与实时更新
通过Qt的`Q_PROPERTY`宏，将C++端的所有比赛数据暴露给QML，数据变化时通过`dataUpdated`信号通知QML自动刷新界面，无需手动调用更新函数，保证了数据与界面的实时同步，完全符合题目「数据实时响应」的要求。

#### 2. 鼠标控制协议适配
针对题目给出的`MouseControl`上行协议，我做了针对性的适配实现：
- 坐标转速度：通过「当前鼠标坐标 - 上一帧坐标」计算相对位移，在固定发送频率下等价于移动速度，完全匹配协议定义
- 方向适配：严格遵循协议要求，x轴负值向左、y轴负值向下、滚轮负值向后滚动
- 按键监听：开启MouseArea的全按键响应，实现左/右/中键的状态监听
- 高频操作节流：对鼠标移动、滚轮、按键操作做了50ms的节流处理，避免操作过快导致的指令泛滥，解决了题目提出的「操作过快」问题

#### 3. 交互与视觉优化
针对题目「关键数据一眼辨别、保留视野完整性」的要求，我做了以下优化：
- 所有UI元素位于界面周围，叠加在图传画面上，不遮挡核心视野
- 低血量血条闪烁，武器温度过高数值变红，操作员无需仔细看数字即可快速识别
- 受击时添加闪红+屏幕抖动动画，还原真实比赛的操作反馈
- 模式选择窗口点击空白处即可关闭，操作流程极简，适配比赛紧张场景

## 项目成长与收获
这个项目的代码部分70%是使用ai写的，我在阅读ai生成的代码的过程中学习QML和c++，理解了后再根据自己想法进行修改或者添加。我发现最难的地方不在于看懂代码，这个多看几遍不懂的问ai一般都能搞懂。最难的是如何根据自己需求写提示词，有时反复问它一直回答不准。
就比如卡我最久的一个是背景图的路径修改，我一开始用本地绝对路径，要改成资源路径"qrc:/cnm/image/sb.png"，明明是复制过来的路径不可能错，然后反复问ai它给出的回答始终是让我在路径上改，卡了一个晚上一直搞不出来，每次运行那底图都是黑的。最后我丢给ai一大段话，才发现是Cmakelist里面有问题，自动生成的一个语句与版本不符，图片没有被编译进去。

第一次做项目，还是收获不少。而且我感觉越做越有兴趣，一开始感觉有点难，还是有点观望的意思，但当我发现构建界面的乐趣后就停不下来了。那个c++部分的代码，一开始完全看不懂，但我一点一点啃，最终理解整个子线程的数据是如何被传到qml界面时（因为那一块一大堆变量名），那种豁然开朗的感觉十分美妙。

还有不足的地方，我觉得一个是对于main文件，头文件里面一些c++定义部分我其实没有完全理解，但这些好像都是比较固定的，时间有限我就专注于能用就好而没有深究底层原理。另外一个就是潜在的 Bug 与改进，比如我注释了子线程对 chassisType （不只这一个）的覆盖，因为懒得一个一个删传递变量，但其实可以花点功夫让代码更优雅。又比如那个受击闪红，每次运行打开画面都会先闪两次，因为不会影响后面，我感觉无伤大雅，就没研究怎么改掉这个小bug。还有红蓝双方的血条 UI 可以用一个 Component 来定义，减少重复代码。
